## Title

## Context / Source
- PR #126 führte die neuen Phase-1 Canonical Contracts ein.
- PR #128 hat den geschlossenen-Candle-Feldnamenkonflikt (`closeTime_ms`) bereits behoben.
- In PR #127 blieben nach Review drei weitere Canonical-Konflikte sichtbar:
  1. `canonical_schema_version` signalisiert mit `4.2.1` einen Patch-Level, obwohl semantische Vertragsänderungen eingeführt wurden.

- die Schema-Versionierung zur dokumentierten SemVer-Policy passt,

- und die UNKNOWN-/Top-K-Budget-Semantik im Liquidity-Stack widerspruchsfrei ist.

## Proposed change (high-level)
Before:
- `canonical_schema_version` bleibt auf `4.2.1`, obwohl die PR semantische Canonical-Vertragsänderungen eingeführt hat.
- `OUTPUT_SCHEMA.md` und `SCORING/SCORE_BREAKOUT_TREND_1_5D.md` definieren widersprüchliche Pflichtfelder pro Output-Row.
- `ORDERBOOK_TOPK_POLICY.md` beschreibt „not fetched (outside Top-K budget)“ noch in einer älteren Rerank-/Output-Semantik, die mit dem neuen `UNKNOWN` stop-path kollidiert.

After:
- `canonical_schema_version` wird auf einen **Major-Bump** angehoben: `5.0.0`.
- `CHANGELOG.md` dokumentiert explizit, dass die Phase-1 Canonical-Neuausrichtung eine breaking canonical contract change ist.
- `OUTPUT_SCHEMA.md` bleibt die kanonische Output-SoT für `trade_candidates`.
- `SCORING/SCORE_BREAKOUT_TREND_1_5D.md` wird so angepasst, dass scorer-spezifische Felder nicht mehr dem globalen Output-Minimum widersprechen.
- `ORDERBOOK_TOPK_POLICY.md` wird an die Phase-1 UNKNOWN-Semantik angepasst:
  - `orderbook_not_in_budget` bleibt ein expliziter `UNKNOWN`-/not-evaluated-Pfad,
  - nicht ein Rerank-/Decision-Pfad für weiterlaufende vollwertige Kandidaten.
- Wo nötig, werden `PIPELINE.md` und `TRADEABILITY_GATE.md` zur Konsistenz nachgezogen.

Edge cases:
- `not fetched (outside Top-K budget)` ≠ `fetched but invalid/malformed/stale`
- scorer-spezifische Berechnungsfelder dürfen weiterhin dokumentiert sein, aber nicht als global verpflichtende `trade_candidates`-Mindestfelder, wenn das Output-Schema sie nicht verlangt
- Major-Versionserhöhung darf nicht in Minor/Patch umgedeutet werden
- bestehende Canonical-Verweise müssen nach der Bereinigung dieselbe Semantik transportieren

Backward compatibility impact:
- Dokumentarisch **breaking** auf Canonical-Ebene.
- `canonical_schema_version` wird bewusst auf `5.0.0` erhöht, um die bereits eingeführten inkompatiblen Semantikänderungen korrekt zu signalisieren.
- Folge-PRs und Implementierungen müssen sich auf die bereinigte Version 5.0.0 beziehen.
