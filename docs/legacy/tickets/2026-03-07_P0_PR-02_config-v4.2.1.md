## Title
[P0] Config V4.2.1 auf das neue Modell heben

## Context / Source
- V4.2.1 ist die alleinige Referenz.
- EPIC 2 / PR-02 verlangt, dass alle neuen Contracts per Config steuerbar sind und keine Magic Numbers im Code verbleiben.
- Die neuen Phase-1-Defaults für Budget, Tradeability, Risk, Decision und BTC-Regime sind in V4.2.1 fachlich vorgegeben.
- `docs/canonical/AUTHORITY.md` verlangt, dass Canonical Vorrang hat und Codex keine fehlende Config-Semantik interpretieren darf.

## Goal
`scanner/config.py` und `config/config.yml` bilden das V4.2.1-Modell vollständig ab, validieren alle neuen Keys deterministisch und bleiben mit bestehenden Configs abwärtskompatibel.

## Proposed change (high-level)
Before:
- Die bestehende Config deckt das neue V4.2.1-Zielmodell nicht vollständig ab.
- Budget-/Tradeability-/Risk-/Decision-/BTC-Regime-Defaults sind noch nicht vollständig oder nicht in der finalen Benennung vorhanden.
- Alte Universe-Filter-Keys sind historisch als harte Gates geprägt und noch nicht sauber als Legacy-Soft-Priors markiert.

After:
- Die Config enthält vollständige, validierte Blöcke für:
  - `budget`
  - `tradeability`
  - `risk`
  - `decision`
  - `btc_regime`
- Alle numerischen Defaults aus V4.2.1 sind zentral dokumentiert und konsistent mit Canonical.
- `pre_shortlist_market_cap_floor_usd` ist korrekt als harter Pool-Guardrail konfigurierbar.
- `universe_filters.market_cap.*` und `universe_filters.volume.*` bleiben kompatibel, sind aber explizit als `legacy_soft_prior` markiert.
- Feature Flags erlauben stufenweises Rollout über:
  - `tradeability.enabled`
  - `risk.enabled`
  - `decision.enabled`
  - `btc_regime.enabled`

Edge cases:
- Missing key vs invalid value sauber unterscheiden
- `enabled: false` deaktiviert den jeweiligen Block deterministisch, ohne den Config-Parser zu brechen
- bestehende Config-Dateien ohne neue Blöcke bleiben lauffähig über Defaults
- keine implizite `bool(...)`-Coercion für Felder, bei denen `null` semantisch relevant sein kann
- `pre_shortlist_market_cap_floor_usd` darf nicht fälschlich als Soft-Prior behandelt werden

Backward compatibility impact:
- Bestehende Configs sollen weiter laden, sofern keine explizit invaliden Werte gesetzt sind.
- Legacy-Keys dürfen nicht still brechen.
- Fehlende neue Keys greifen auf zentrale Defaults zurück; invalide Werte erzeugen klaren Fehler.

## Codex Implementation Guardrails (No-Guesswork, Pflicht bei Code-Tickets)
> Dieses Ticket ist ein Code-Ticket. Kein Raten. Fehlende Semantik muss explizit im Ticket oder Canonical stehen.

- **Zentrale Defaults verwenden:** Keine raw-dict-Defaults an Call Sites. Config-Lesen und Validieren muss dieselben zentralen Defaults nutzen wie `ScannerConfig` bzw. die bestehende zentrale Config-Accessor-Logik.
- **Missing ≠ invalid:** Fehlender Key darf auf dokumentierten Default fallen; vorhandener aber invalider Wert muss klar fehlschlagen.
- **Keine stillen Fallbacks:** Wenn ein Wert vorhanden, aber semantisch ungültig ist, klarer Fehler statt stiller Korrektur.
- **Keine bool()-Coercion für Nullable-Semantik:** Falls ein Feld optional/nullable ist, muss `None` als `null` erhalten bleiben und darf nicht implizit zu `False` werden.
- **Feature Flags deterministisch:** `enabled: false` muss zu reproduzierbarem Verhalten führen; keine impliziten Nebeneffekte.
- **Legacy-Keys nicht heimlich entfernen:** Weiterhin lesbar halten oder sauber als deprecated/legacy markieren, ohne inkompatiblen Bruch.
- **Benennung exakt gemäß V4.2.1:** Key muss `pre_shortlist_market_cap_floor_usd` heißen, nicht `soft_market_cap_floor_usd`.
- **BTC-Regime Mode validieren:** In Phase 1 ist `threshold_modifier` der kanonische Modus; unbekannte Modi sind invalid.
- **Class thresholds validieren:** Schwellenwerte für `DIRECT_OK`, `TRANCHE_OK`, `MARGINAL` müssen vorhanden und sinnvoll geordnet sein.

## Implementation Notes (optional but useful)
Die neue Config muss mindestens folgende Blöcke und Defaults abbilden:

```yaml
budget:
  shortlist_size: 200
  orderbook_top_k: 200
  pre_shortlist_market_cap_floor_usd: 25000000

tradeability:
  enabled: true
  notional_total_usdt: 20000
  notional_chunk_usdt: 5000
  max_tranches: 4
  band_pct: 1.0
  max_spread_pct: 0.15
  min_depth_1pct_usd: 200000
  class_thresholds:
    direct_ok_max_slippage_bps: 50
    tranche_ok_max_slippage_bps: 100
    marginal_max_slippage_bps: 150

risk:
  enabled: true
  stop_method: atr_multiple
  atr_period: 14
  atr_timeframe: 1d
  atr_multiple: 2.0
  min_stop_distance_pct: 4.0
  max_stop_distance_pct: 12.0
  min_rr_to_tp10: 1.3

decision:
  enabled: true
  min_score_for_enter: 65
  min_score_for_wait: 40
  require_tradeability_for_enter: true
  require_risk_acceptable_for_enter: true

btc_regime:
  enabled: true
  mode: threshold_modifier
  risk_off_enter_boost: 15
```

Zusätzlich:
- `universe_filters.market_cap.*` und `universe_filters.volume.*` als `legacy_soft_prior` markieren
- Defaults in `config/config.yml`
- Python-seitige Validierung in `scanner/config.py`
- Dokumentierte Merge-/Override-Semantik darf nicht auseinanderlaufen
