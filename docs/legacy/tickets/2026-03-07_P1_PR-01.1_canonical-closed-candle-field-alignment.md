## Title
[P1] Ticket 1.1 — Canonical closed-candle field name in PIPELINE.md auf `closeTime_ms` angleichen

## Context / Source
- Ticket 1 / PR-01 hat die kanonischen Verträge für die Neuausrichtung festgezogen.
- Nach Umsetzung hat Codex einen Canonical-Conflict identifiziert:
  - `docs/canonical/DATA_SOURCES.md` definiert die Closed-Candle-Policy über `closeTime_ms <= asof_ts_ms`.
  - `docs/canonical/PIPELINE.md` beschreibt an mindestens einer Stelle den Closed-Candle-Gate mit einer abweichenden Feldbezeichnung (`close_time` bzw. sinngleiche nicht-kanonische Variante).
- Laut `docs/canonical/AUTHORITY.md` darf es keine widersprüchlichen kanonischen Wahrheiten geben.

## Goal
Die kanonische Closed-Candle-Policy ist in allen betroffenen Canonical Docs widerspruchsfrei auf dieselbe Feldsemantik ausgerichtet:
- Raw-Provider-Feld: `closeTime`
- kanonisch normalisiertes Feld: `closeTime_ms`
- Closed-Candle-Gate: `closeTime_ms <= asof_ts_ms`

## Proposed change (high-level)
Before:
- `DATA_SOURCES.md` verwendet kanonisch `closeTime_ms`.
- `PIPELINE.md` verwendet an mindestens einer Stelle eine abweichende Benennung für denselben Gate.
- Dadurch können Consumer zwei inkompatible Canonical-Anweisungen lesen.

After:
- `PIPELINE.md` verwendet dieselbe kanonische Feldbezeichnung wie `DATA_SOURCES.md`.
- Falls zwischen Raw-Provider-Feld und kanonisch normalisiertem Feld unterschieden wird, ist die Beziehung explizit und widerspruchsfrei dokumentiert:
  - Raw: `closeTime`
  - Canonical normalized: `closeTime_ms`
- Der Closed-Candle-Gate wird nur noch als `closeTime_ms <= asof_ts_ms` beschrieben.

Edge cases:
- Nicht mit `close_time`, `close_ts`, `candle_close_time` oder ähnlichen Aliasen mischen
- Falls in `PIPELINE.md` sowohl Raw- als auch normalisierte Felder erwähnt werden, muss klar sein, welches Feld für Vergleiche verwendet wird
- Keine implizite Annahme, dass ein Consumer das richtige Feld „schon meinen wird“

Backward compatibility impact:
- Rein dokumentarische Präzisierung / Bugfix im Canonical
- Nachfolgende Tickets und Implementierungen müssen ausschließlich die kanonische Benennung verwenden
